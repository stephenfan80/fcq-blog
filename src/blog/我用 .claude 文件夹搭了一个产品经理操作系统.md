三个月前我开始用 Claude Code，最初和大多数人一样——打开终端，问问题，复制答案。

转折点是我接了一个真实项目。

朋友在哈尔滨开了一家叫"闲逛"的生活方式集合空间，咖啡加家居加艺术。她自己硬扛小红书运营，没时间也不专业。我想帮她，但我自己也不是新媒体运营出身。

于是我拿 Claude Code 来做这件事。结果发现，**当你用 AI 做一个真实的、有截止日期的、要交付给别人的项目时，"聊天"这个模式瞬间就崩了。**

## 聊天模式为什么崩了

第一次打开 Claude Code 准备分析闲逛的小红书数据时，我发现自己要做一大堆重复的事：

每次新对话都要解释一遍项目背景——这是一家什么店、在哪个城市、目标是什么。每次都要说一遍数据在哪个目录、图片存在哪里。每次都要提醒它用中文输出、不要给我没有数据支撑的结论。

更麻烦的是品牌调性。闲逛不是网红奶茶店，它的调性是"慢"和"生活感"。我第一次忘了说这个，Claude 给我生成了一堆带感叹号和 emoji 的小红书文案。完全不对。

一个对话解决一个问题没问题。但当你在同一个项目里持续工作几周，**每次都从零开始教 AI "你是谁"，这件事本身就变成了最大的效率瓶颈。**

## 第一个顿悟：把"每次都要说的话"固化下来

后来我读到了 `.claude/` 文件夹的用法。第一反应是：这不就是我需要的吗？

我在闲逛的项目目录下建了一个 `CLAUDE.md`，把所有上下文写进去：

- 闲逛是什么——哈尔滨生活方式集合空间，产品线涵盖咖啡、家居买手、艺术展览
- 数据在哪里——原始数据路径、图片路径、分析报告路径
- 怎么工作——中文输出、结论要有数据支撑、不要过度美化

然后在 `.claude/rules/` 里建了几个规则文件。品牌调性单独一个文件，小红书分析规范一个文件，素材管理规范一个文件。品牌调性那个文件里写着：闲逛的核心调性是"慢下来的时间感"，禁用感叹号堆砌，禁用网红腔，禁用"绝绝子""yyds"。

效果立竿见影。**从那以后，每次打开这个项目的 Claude Code，它已经知道自己在服务谁、数据在哪里、调性是什么。** 我不再需要重复解释任何背景信息，直接开始干活。

一个小小的文件夹配置，省掉了每天十几分钟的重复沟通。三个星期下来就是好几个小时。

## 第二个顿悟：把重复的工作流变成一条命令

闲逛的项目里，有一个流程我每周都要做：拉取最新的小红书数据，分析互动趋势，看看哪些内容类型表现好，然后生成一份周报。

前几次我都是手动一步步引导 Claude 完成。每次大概要花 20 分钟的来回对话。

后来我把这个流程封装成了一个 skill——一个叫 `/weekly-report` 的斜杠命令。一敲命令，Claude 自动读数据、做分析、生成周报。20 分钟变成 2 分钟。

做完闲逛的 skill 之后我上了瘾。我开始给自己的其他工作流也做 skill：

写播客转录的——我经常听播客，想把内容转成文字存到 Obsidian 笔记里，于是做了一个 `/podcast-transcribe`，输入一个音频文件，输出带说话人标注的对话文本。

写小红书采集的——帮闲逛做竞品分析需要批量抓取小红书数据，于是做了 `/xhs-scraper`，输入一个账号 URL，输出全部帖子的标题、正文、互动数据和图片。

写 Obsidian 发公众号的——我在 Obsidian 里写东西，想一键发到微信公众号，于是做了 `/obsidian-to-wechat`。

到今天，我的 Claude Code 里装了 86 个 skill。不是为了凑数，每一个都是因为我在做某个项目时发现"这个动作我已经重复了第三遍"。

**Skill 的本质不是"快捷命令"。它是把你的工作经验编码成了一套可重复执行的流程。** 你教 Claude 做一次，它就永远会做了。这不就是带新人吗？只不过这个新人过目不忘，而且不会离职。

## 第三个顿悟："希望它做到"和"必须它做到"是两件事

闲逛项目做到第二周，出了一次事故。

我在 CLAUDE.md 里写了"不要自动提交 git commit"。但在一次很长的对话中，Claude 帮我改完代码后，顺手提交了。它不是故意的——它只是在长上下文中忘了这条规则。

这件事让我意识到一个重要的区别：

**CLAUDE.md 里写的东西是"期望"——你希望 Claude 这样做，它大多数时候会听，但不是保证。**

**settings.json 里写的东西是"强制"——客户端层面的硬约束，Claude 物理上做不到。**

从那以后，我把真正不能出错的规则都写在 settings.json 里：哪些命令可以执行、哪些文件不能读、哪些域名可以访问。写在 CLAUDE.md 里的都是"建议"——工作风格偏好、输出格式要求、分析方法论。

这两者混在一起是很多人踩的坑。你把安全相关的规则写在 CLAUDE.md 里，就像把门禁密码写在便利贴上贴在门口——大多数时候有用，但没有保障。

还有一个容易忽略的盲区：`Read(./.env)` 的 deny 规则只约束 Read 工具，不约束 Bash。如果你同时允许了 Bash，Claude 用 `cat .env` 照样能读到你的密钥。

## 三个月后回头看

从闲逛开始，到后来做拾读（一个 AI 内容策展项目）、做博客系统、做各种数据分析，三个月下来，我对 Claude Code 的理解完全变了。

最大的变化不是技术层面的。是心态。

一开始我把 Claude Code 当一个"很聪明的助手"——我问它问题，它给我答案。交互模式是一问一答，每次都是一次性的。

现在我把它当一个"可以被配置的系统"——我投入时间搭建项目配置、封装工作流、设定规则边界，然后它在这个系统里高效运转。交互模式是建设性的，每一次配置都在积累。

这个转变的代价是什么？一个项目的 `.claude/` 文件夹，从零到可用，大概需要半天时间。写 CLAUDE.md、拆 rules、做第一个 skill。

回报是什么？**之后每一天，你和 AI 的协作效率都比昨天高一点。** 因为昨天踩过的坑变成了今天的规则，昨天重复的操作变成了今天的 skill，昨天手动交代的上下文变成了今天自动加载的背景知识。

这就是"聊天"和"搭系统"的区别。聊天是一次性的，聊完就没了。搭系统是积累性的，用得越多越顺手。

> 和 AI 协作最大的杠杆不在于你问了一个多好的问题，而在于你搭了一个多好的系统。
